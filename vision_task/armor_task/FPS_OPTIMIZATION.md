# 帧率优化记录

本文记录本次围绕 `armor_task` 与 `hiki_ros2` 做过的提帧与稳帧改动，方便后续回溯、复现实验结果和继续优化。

## 1. `video_test` 侧改动

### 1.1 去掉人为限帧
- 删除固定目标帧率等待逻辑，不再按目标 FPS 播放
- 改为 `cv::waitKey(1)`，直接观察实时处理帧率

### 1.2 降低无效开销
- 删除循环中的大量 `std::cout` 调试输出
- 去掉显示前的整帧 `clone()`，改为直接在当前帧上绘制

### 1.3 增加分级绘制
- 新增 `DrawMode`：`OFF / LITE / FULL`
- 按 `v` 可切换绘制等级
- `OFF` 模式下尽量接近纯处理路径，便于区分“算法瓶颈”和“显示瓶颈”

### 1.4 增加更细的阶段统计
- 增加 `detect / track / aim / draw / total` 阶段耗时显示
- 便于定位每一帧真正耗时集中在哪一段

### 1.5 读帧与处理解耦
- 增加读视频线程
- 主线程总是消费最新帧，避免 `cap.read()` 与检测串行绑定

## 2. `auto_aimer_test` 侧改动

### 2.1 去掉影响实时性的日志
- 删除命令发送线程高频日志
- 删除 IMU 姿态高频日志
- 删除部分 target 初始化和调试输出

### 2.2 无显示器时禁用 OpenCV 窗口
- 自动检测是否存在显示环境
- 无显示器时不执行 `cv::imshow()` / `cv::waitKey()`

### 2.3 增加分级绘制与阶段统计
- 同样支持 `OFF / LITE / FULL`
- 同样显示 `detect / track / aim / draw / total`

### 2.4 图像处理改成最新帧覆盖模型
- 新增独立取图线程
- 主处理线程只消费最新图像，不堆积旧帧
- 降低实时流处理中的延迟累积

### 2.5 减少图像拷贝
- 图像线程和处理线程之间不再传递 `cv::Mat` 深拷贝
- 改为共享 `FramePacket` 指针

## 3. `ROS2Manager` 侧改动

### 3.1 订阅队列改为实时优先
- 使用 `SensorDataQoS().keep_last(1)`
- 避免旧帧积压导致延迟上升

### 3.2 增加帧 ID 管理
- 使用帧 ID 跟踪“最新帧 / 已消费帧”
- 确保每帧只消费一次

### 3.3 减少图像传递中的重复 `clone`
- 回调中生成 `FramePacket`
- 对外提供 `get_frame_packet()`
- 让下游通过共享指针拿到最新帧包，而不是再次深拷贝

## 4. 检测与跟踪侧改动

## 4.1 `tracker` 改动
- 去掉一部分高频调试输出
- 初始目标选择仍以“图像中心附近目标”为主
- 更新阶段由“首个匹配就更新”改为：
  - 对同 `car_num` 候选做 PnP
  - 与 EKF 预测结果做匹配打分
  - 只选择最优观测更新滤波器
- 目标：兼顾性能与 EKF 稳定性，避免粗暴优化导致跟踪精度下降

### 4.2 `detector` 改动
- 删除每帧检测调试输出
- 后处理容器提前 `reserve`
- 在 CPU 侧封装前增加 `Top-K` 截断，减少后续处理量

## 5. YOLO / TensorRT 后处理改动

### 5.1 去掉 CPU NMS
- 原先使用 `cv::dnn::NMSBoxes(...)`
- 现已改为 GPU 侧 `decode + NMS + compact`

### 5.2 减少同步点
- 调整 `decode/NMS` 之后的数据回传方式
- 减少不必要的 GPU/CPU 同步等待

### 5.3 当前状态
- 现在是 `TensorRT + CUDA 后处理`
- 还不是把 NMS plugin 完全并进 engine 的“纯 TensorRT engine 内部后处理”
- 但已经去掉了 CPU NMS 这部分明显瓶颈

## 6. `hiki_ros2` / `hik_camera_node` 改动

### 6.1 发布格式改为 `bgr8`
- 与下游 `cv_bridge::toCvShare(msg, "bgr8")` 对齐
- 避免下游再做一次颜色空间转换

### 6.2 预分配图像缓冲
- 启动时一次性分配图像缓冲
- 避免每帧 `resize` 带来的额外开销

### 6.3 增加采集阶段 profiling
- 每秒打印一次：
  - `get`
  - `convert`
  - `publish`
  - `capture fps`
- 用于区分瓶颈在 SDK 取帧、像素转换还是 ROS 发布

### 6.4 增加 image-only 模式
- 支持只发布 `sensor_msgs::msg::Image`
- 用于验证 `camera_info` 是否是性能瓶颈
- 实测表明 `camera_info` 不是主要瓶颈

### 6.5 探测并尝试使用原生彩色输出
- 启动时读取相机 `PixelFormat`
- 优先尝试切到相机原生 `BGR8`
- 若支持，则绕过 `MV_CC_ConvertPixelType(...)`
- 实测 `convert` 从约 `4.8ms` 降到约 `0.5ms`

### 6.6 支持分辨率与 ROI 参数化
- 新增参数：
  - `width`
  - `height`
  - `offset_x`
  - `offset_y`
- 可直接通过 launch 设置，不必每次在 MVS 手工调整

## 7. 当前定位出的主要瓶颈变化

### 7.1 优化前
- 人为限帧
- 高频日志
- 绘制开销
- 多次整帧 `clone`
- CPU NMS
- 相机节点像素转换

### 7.2 优化后
- `video_test` 与 `auto_aimer_test` 的显示与日志开销明显下降
- `hik_camera_node` 的 `convert` 已显著下降
- 当前新的关键瓶颈更多落在：
  - 相机原生 `BGR8` 带来的取帧/传输耗时
  - 整体链路中的剩余内存搬运与调度
  - 端到端处理总链路，而不再是单一模块

## 8. 已验证结论

- 相机节点本身可达到约 `97~98 FPS`
- `camera_info` 不是主要瓶颈
- `auto_aimer_test` 无显示模式下已避免 GUI 开销
- 当前系统仍有继续提帧空间，但需要继续在“相机输出格式 / 分辨率 / 下游链路搬运成本”之间做平衡

## 9. 后续建议

### 优先级较高
- 继续测试更低分辨率或更小 ROI
- 对比“原生 BGR8”与“原始 Bayer + 下游转换”两条路径
- 继续减少端到端图像搬运次数

### 可选方向
- 将 NMS 进一步并入 TensorRT engine
- 继续拆分处理线程与显示线程
- 做更细的端到端时间统计：图像到达间隔、等待新帧时间、处理时间
